Token Payload (HMAC-SHA256 signed)

          
{
  "userId": "alice",
  "deviceId": "alice_pixel_8",
  "issuedBalancePaise": 50000,       // ₹500
  "offlineLimitPaise": 500000,        // ₹5000 — offline cap
  "offlineRemainingPaise": 500000,
  "counter": 5,                      // monotonically increasing
  "expiresAt": "2025-05-03T00:00Z",  // 24h TTL
  "issuedAt": "2025-05-02T00:00Z",
  "sessionPublicKey": "base64...",   // ECDSA pubkey for receipts
  "hmac": "base64_sha256..."         // signs all above fields
}

        


        

          
NFC Tap Protocol (3 messages)

          
// 1. Bob → Alice (nonce challenge)
{
  nonce: "a7f3b9c2...",       // SecureRandom, 32 bytes
  receiverDeviceId: "bob_phone"
}

// 2. Alice → Bob (signed payment)
{
  payerUserId: "alice",
  payerDeviceId: "alice_pixel_8",
  receiverDeviceId: "bob_phone",
  amountPaise: 5000,           // ₹50
  counter: 6,                  // incremented
  nonce: "a7f3b9c2...",        // echoed back
  tokenExpiresAt: "...",
  hmac: "..."                  // HMAC over all above
}

// 3. Bob → Alice (mutual receipt)
{
  receivedPaise: 5000,
  fromCounter: 6,
  payerUserId: "alice",
  receiverDeviceId: "bob_phone",
  nonce: "a7f3b9c2...",
  sig: "ecdsa_sig..."          // Bob's Keystore key
}

        


        

          
Production Upgrade Path

          

            

              
HMAC → ECDSA (secp256k1)

              
HMAC is symmetric — only the server can verify. ECDSA lets receiver phones verify offline without the signing key. Identical to how Bitcoin lets any node verify a transaction. EMV uses ECDSA for the same reason.

              
Documented

            

            

              
Android Keystore → StrongBox

              
StrongBox provides hardware-bound keys with setMaxUsageCount(n), making keys non-exportable and physically isolated in a secure element. Skipped due to manufacturer fragmentation (Samsung vs Pixel vs others).

              
Documented

            

            

              
Device Attestation

              
Google Play Integrity API (or SafetyNet) verifies the device hasn't been rooted and the app hasn't been tampered with. Needed before issuing high-limit tokens in production.

              
Documented

            

          

        


        

          
The Interview Answer

          

            "The ₹200 limit is RBI's regulatory choice for e-Rupee, not a technical constraint. My system uses a tiered limit based on KYC level — higher verification unlocks higher offline spending. The nonce binding and mutual receipt system means any double-spend attempt is cryptographically traceable to the payer, which justifies a higher limit than anonymous token systems. The limit is a risk dial, not a ceiling."
